§ 01What is AOP?

Aspect-Oriented Programming (AOP) is a programming paradigm that aims to increase modularity by allowing the separation of cross-cutting concerns. It adds behavior to existing code without modifying the code itself.

The Core Problem: Cross-Cutting Concerns

In any real application, certain behaviors cut across multiple modules — they are needed everywhere but don't belong to any specific business logic. These are called cross-cutting concerns:

  • Logging — every method needs to log entry/exit
  • Security — every endpoint needs authorization checks
  • Transaction management — every DB operation needs commit/rollback
  • Caching — results need to be cached transparently
  • Performance monitoring — measure how long methods take
  • Auditing — track who did what and when

Why OOP Alone is Insufficient

Object-Oriented Programming is excellent at encapsulating state and behavior into objects. But it struggles when the same concern needs to appear in dozens of unrelated classes. OOP gives us inheritance and interfaces, but these can't solve the problem cleanly.

Why This is a Disaster

  • Violation of DRY — Don't Repeat Yourself. The same 5 lines are copied in every method.
  • Hard to change — If you want to change log format, you update 50+ methods.
  • Business logic is buried — The actual 1-2 lines of logic are hidden in boilerplate.
  • Testing nightmare — Unit tests now have to deal with logging side effects.
  • Risk of forgetting — New developers adding new services will forget to add logging.
⚠️ The Real Cost
In a production app with 100 service methods, that's 500+ lines of pure boilerplate. Change the logging format? 100 manual updates. Miss one? Inconsistent logs in production. AOP solves this entirely.

§ 03Core AOP Concepts

Before writing any code, you must deeply understand the 7 core concepts. Each concept has a precise meaning — confusing them in interviews is a common mistake.

1. Aspect

An Aspect is the modularization of a cross-cutting concern. It's a regular Java class annotated with @Aspect that contains all the advice (behavior) for a particular concern.

🔍 Analogy
Think of an Aspect as a security guard company. The company (Aspect) defines all the rules about what to check and how to respond. Individual guards (Advice) execute those rules at specific locations (JoinPoints).

2. Advice

Advice is the actual action taken by an Aspect at a particular JoinPoint. It is the code that runs. There are 5 types of advice:

  • @Before — runs before the method
  • @After — runs after the method (always, like finally)
  • @AfterReturning — runs after successful return
  • @AfterThrowing — runs after exception is thrown
  • @Around — wraps the method, most powerful

3. JoinPoint

A JoinPoint is a specific point in program execution where advice can be applied. In Spring AOP, a JoinPoint is always a method execution. In full AspectJ, it could be field access, constructor calls, etc.

🔍 Analogy
JoinPoints are like specific intersections in a city where traffic cameras (advice) can be placed. The cameras don't exist on every inch of road — only at defined intersections.

4. Pointcut

A Pointcut is an expression that selects which JoinPoints will have advice applied. It's the filter that says "apply my advice to these methods".

Pointcut Examples
// Apply to ALL methods in UserService
@Pointcut("execution(* com.example.service.UserService.*(..))")
public void userServiceMethods() {}

// Apply to ALL methods in ALL classes in the service package
@Pointcut("execution(* com.example.service.*.*(..))")
public void allServiceMethods() {}

// Apply to methods annotated with @LogExecutionTime
@Pointcut("@annotation(com.example.annotation.LogExecutionTime)")
public void loggedMethods() {}

Weaving is the process of linking Aspects with other application types or objects to create an advised object. Spring AOP does weaving at runtime using proxies.

6. Target Object

The Target Object is the original object being advised — your actual service class. The proxy wraps around it and delegates calls to it after applying advice.

7. Proxy

The Proxy is the object created by Spring to implement the Aspect contracts. When you autowire a service, you actually get the proxy, not the real object. The proxy intercepts calls and applies advice before/after delegating to the real object.

Two Types of Proxies

Spring AOP supports two proxy mechanisms. Understanding when each is used is critical for debugging AOP issues.

JDK Dynamic Proxy

Used when the target object implements at least one interface. Spring creates a proxy that implements the same interface(s) and delegates calls to the target.

JDK Dynamic Proxy — Interface-based
// Interface exists
public interface UserService {
    User createUser(String name);
}

// Implementation
@Service
public class UserServiceImpl implements UserService {
    public User createUser(String name) { ... }
}

// Spring creates: Proxy implements UserService, delegates to UserServiceImpl
// Type: java.lang.reflect.Proxy

CGLIB Proxy

Used when the target object does NOT implement an interface (or when proxyTargetClass=true). CGLIB (Code Generation Library) generates a subclass of the target class at runtime and overrides all non-final methods.

CGLIB Proxy — Subclass-based
// No interface — CGLIB is used
@Service
public class OrderService {
    public Order createOrder(Long userId) { ... }
}

// CGLIB creates: class OrderService$$EnhancerBySpringCGLIB extends OrderService
// Overrides createOrder() to apply advice
// proxyTargetClass = true forces CGLIB even with interfaces

LoggingAspect.java — All Logging in One Place
@Aspect
@Component
@Slf4j
public class LoggingAspect {

    // Pointcut: all public methods in the service package
    @Pointcut("execution(public * com.example.service.*.*(..))")
    public void serviceLayer() {}

    // Before Advice: runs before every service method
    @Before("serviceLayer()")
    public void logMethodEntry(JoinPoint joinPoint) {
        String className = joinPoint.getTarget().getClass().getSimpleName();
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();
        log.info("==> {}.{}() called with args: {}", className, methodName, Arrays.toString(args));
    }

    // After Returning: runs after successful execution
    @AfterReturning(pointcut = "serviceLayer()", returning = "result")
    public void logMethodExit(JoinPoint joinPoint, Object result) {
        String methodName = joinPoint.getSignature().getName();
        log.info("<== {}() returned: {}", methodName, result);
    }

    // After Throwing: runs on exception
    @AfterThrowing(pointcut = "serviceLayer()", throwing = "ex")
    public void logException(JoinPoint joinPoint, Exception ex) {
        String methodName = joinPoint.getSignature().getName();
        log.error("❌ {}() threw: {}", methodName, ex.getMessage());
    }
}

@Before — Complete Examples
@Aspect
@Component
@Slf4j
public class BeforeExamples {

    // ─── Example 1: Method entry logging ───
    @Before("execution(* com.example.service.*.*(..))")

            jp.getSignature().getName());
    }

    // ─── Example 2: Input validation ───
    @Before("execution(* com.example.service.UserService.createUser(..))")
    public void validateUserInput(JoinPoint jp) {
        Object[] args = jp.getArgs();
        User user = (User) args[0];
        if (user.getEmail() == null || user.getEmail().isBlank()) {

    @Before("execution(* com.example.service.AdminService.*(..))")
    public void checkAdminRole(JoinPoint jp) {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (!auth.getAuthorities().contains(new SimpleGrantedAuthority("ROLE_ADMIN"))) {
            throw new AccessDeniedException("Admin role required");
        }
    }
}

@AfterThrowing — Exception Monitoring
@Aspect
@Component
@Slf4j
public class ExceptionMonitoringAspect {

    // ─── Example 1: Log all service exceptions ───
    @AfterThrowing(
        pointcut = "execution(* com.example.service.*.*(..))",
        throwing = "ex"  // Binds to the thrown exception
    )
    public void logServiceException(JoinPoint jp, Exception ex) {
        String method = jp.getSignature().toShortString();
        log.error("Exception in {}: [{}] {}", method, ex.getClass().getSimpleName(), ex.getMessage());
    }

    // ─── Example 2: Alert on specific exceptions ───
    @AfterThrowing(
        pointcut = "execution(* com.example.service.PaymentService.*(..))",
        throwing = "ex"
    )
    public void alertOnPaymentFailure(JoinPoint jp, RuntimeException ex) {
        // Only fires for RuntimeException and subclasses
        String method = jp.getSignature().getName();
        alertingService.sendAlert(
            "PAYMENT_FAILURE",
            "Method: " + method + " | Error: " + ex.getMessage()
        );
    }

    // ─── Example 3: Track error metrics ───
    @AfterThrowing(
        pointcut = "within(com.example..*)",
        throwing = "ex"
    )
    public void trackErrorMetrics(JoinPoint jp, Throwable ex) {
        String className = jp.getTarget().getClass().getSimpleName();
        metricsRegistry.counter("errors",
            "class", className,
            "exception", ex.getClass().getSimpleName()
        ).increment();
    }
}
📌 Exception Filtering
The throwing parameter type acts as a filter. If you declare RuntimeException ex, the advice only fires for RuntimeException and subclasses. Declare Throwable ex to catch everything.

§ 11@Around Advice

@Around is the most powerful advice type. It completely wraps the method execution. You control whether the method runs at all, can modify arguments, can change the return value, and can catch exceptions. Use it carefully.

@Around ends

You can skip proceed() to never call the method, or call it multiple times (retry)

@Around — Complete Examples
@Aspect
@Component
@Slf4j
public class AroundExamples {

    // ─── Example 1: Execution Time Measurement ───
    @Around("execution(* com.example.service.*.*(..))")
    public Object measureExecutionTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        
        try {
            Object result = pjp.proceed(); // Call the actual method
            long elapsed = System.currentTimeMillis() - start;
            log.info("[PERF] {}.{}() completed in {}ms",
                pjp.getTarget().getClass().getSimpleName(),
                pjp.getSignature().getName(),
                elapsed);
            return result; // MUST return the result!
            
        } catch (Exception ex) {
            long elapsed = System.currentTimeMillis() - start;
            log.error("[PERF] {}.{}() FAILED in {}ms: {}",
                pjp.getTarget().getClass().getSimpleName(),
                pjp.getSignature().getName(),
                elapsed, ex.getMessage());
            throw ex; // Re-throw! Don't swallow exceptions silently
        }
    }

    // ─── Example 2: Retry Mechanism ───
    @Around("@annotation(com.example.annotation.Retryable)")
    public Object retryOnFailure(ProceedingJoinPoint pjp) throws Throwable {
        int maxRetries = 3;
        int attempt = 0;
        
        while (attempt < maxRetries) {
            try {
                return pjp.proceed(); // Try to execute
            } catch (TransientException ex) {
                attempt++;
                if (attempt >= maxRetries) throw ex;
                log.warn("Attempt {} failed, retrying: {}", attempt, ex.getMessage());
                Thread.sleep(1000L * attempt); // Exponential-ish backoff
            }
        }
        throw new RuntimeException("Should not reach here");
    }

    // ─── Example 3: Modify Arguments ───
    @Around("execution(* com.example.service.UserService.createUser(..))")
    public Object sanitizeInput(ProceedingJoinPoint pjp) throws Throwable {
        Object[] args = pjp.getArgs();
        User user = (User) args[0];
        
        // Sanitize input before passing to method
        user.setEmail(user.getEmail().toLowerCase().trim());
        user.setName(StringUtils.capitalize(user.getName().trim()));
        
        return pjp.proceed(args); // Pass modified args
    }

    // ─── Example 4: Cache ───
    @Around("execution(* com.example.service.ProductService.getProduct(..))")
    public Object cacheResult(ProceedingJoinPoint pjp) throws Throwable {
        Long productId = (Long) pjp.getArgs()[0];
        String cacheKey = "product:" + productId;
        
        Object cached = cache.get(cacheKey);
        if (cached != null) {
            log.debug("Cache HIT for {}", cacheKey);
            return cached; // Return from cache, skip actual method!
        }
        
        Object result = pjp.proceed(); // Cache miss — call the real method
        cache.put(cacheKey, result);
        return result;
    }
}
⚠️ Critical Rules for @Around
  1. Always return the result of pjp.proceed(). Returning null silently breaks the caller.
  2. Always rethrow exceptions unless you explicitly want to swallow them.
  3. The method signature must declare throws Throwable.
  4. Use ProceedingJoinPoint, not JoinPoint — only PJP has proceed().

§ 12Pointcut Expressions Deep Dive

Pointcut expressions are the most critical skill in Spring AOP. Writing them correctly determines which methods get intercepted. Mastering the syntax separates beginner from senior-level AOP usage.

Retry Mechanism with @Around

Production Retry with Exponential Backoff
@Aspect
@Component
@Slf4j
public class RetryAspect {

    @Around("@annotation(retryable)")
    public Object retry(ProceedingJoinPoint pjp, Retryable retryable) throws Throwable {
        int maxAttempts = retryable.maxAttempts();      // From annotation
        long delayMs = retryable.delayMs();             // Initial delay
        Class<?>[] retryOn = retryable.retryOn();      // Exception types

        Throwable lastException = null;

        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                return pjp.proceed();
            } catch (Throwable ex) {
                boolean shouldRetry = Arrays.stream(retryOn)
                    .anyMatch(c -> c.isAssignableFrom(ex.getClass()));

                if (!shouldRetry || attempt >= maxAttempts) throw ex;

                lastException = ex;
                long wait = delayMs * (long) Math.pow(2, attempt - 1); // Exponential backoff
                log.warn("[Retry {}/{}] {} failed: {}. Retrying in {}ms...",
                    attempt, maxAttempts,
                    pjp.getSignature().getName(),
                    ex.getMessage(), wait);
                
                Thread.sleep(wait);
            }
        }
        throw lastException;
    }
}

    public CompletableFuture<Void> sendEmail(String to, String body) {
        // Runs in a background thread
        emailSender.send(to, body);
        return CompletableFuture.completedFuture(null);
    }
}

// Internal Flow:
// 1. Caller calls sendEmail() on proxy
// 2. AsyncExecutionInterceptor intercepts
// 3. Wraps method call in Callable/Runnable
// 4. Submits to TaskExecutor thread pool
// 5. Returns CompletableFuture immediately to caller
// 6. Method runs asynchronously in background thread
⚠️ @Async Self-Invocation Bug
Same as @Transactional — calling an @Async method from within the same class bypasses the proxy and runs synchronously. Always inject the bean or call from a different class.

§ 21Common Mistakes in Spring AOP

Mistake 1: Self-Invocation

Self-Invocation — AOP is Bypassed
// ❌ BROKEN — self-invocation bypasses proxy
@Service
public class UserService {
    @LogExecutionTime
    public void doA() {
        this.doB(); // Calls real object directly — @LogExecutionTime on doB() is IGNORED
    }

    @LogExecutionTime
    public void doB() { ... }
}

// ✅ FIX 1: Use ApplicationContext to get the proxy
@Service
public class UserService implements ApplicationContextAware {
    private ApplicationContext ctx;
    
    public void doA() {
        ctx.getBean(UserService.class).doB(); // Gets the proxy
    }
}

// ✅ FIX 2: Inject self (Spring 4.3+)
@Service
public class UserService {
    @Autowired
    @Lazy // Important to avoid circular dependency
    private UserService self;

    public void doA() {
        self.doB(); // Goes through proxy
    }
}

Mistake 2: Final Methods

Final Method — CGLIB Cannot Override
// ❌ BROKEN — CGLIB cannot subclass and override final methods
@Service
public class PaymentService {
    @Transactional // IGNORED! Final method cannot be proxied
    public final void processPayment(Long amount) { ... }
}

// ✅ FIX: Remove 'final'
@Service
public class PaymentService {
    @Transactional
    public void processPayment(Long amount) { ... } // Works!
}

Mistake 5: Wrong Pointcut — Forgetting Subpackages

Mistake 6: Not Returning proceed() Result

Silently Discarding Return Values
// ❌ BROKEN — returns null for all methods!
@Around("execution(* com.example.service.*.*(..))")
public Object trace(ProceedingJoinPoint pjp) throws Throwable {
    pjp.proceed(); // Return value discarded!
    return null;   // All method callers get null
}

// ✅ FIX
@Around("execution(* com.example.service.*.*(..))")
public Object trace(ProceedingJoinPoint pjp) throws Throwable {
    Object result = pjp.proceed();
    return result; // Always return the result!
}

Mistake 7: Swallowing Exceptions

Silent Exception Swallowing
// ❌ BROKEN — hides all exceptions from callers
@Around("execution(* com.example.service.*.*(..))")
public Object safeWrapper(ProceedingJoinPoint pjp) throws Throwable {
    try {
        return pjp.proceed();
    } catch (Exception ex) {
        log.error("Error: " + ex.getMessage());
        return null; // Exception swallowed! Caller thinks it succeeded
    }
}

// ✅ FIX — log AND rethrow
@Around("...")
public Object safeWrapper(ProceedingJoinPoint pjp) throws Throwable {
    try {
        return pjp.proceed();
    } catch (Exception ex) {
        log.error("Error: " + ex.getMessage());
        throw ex; // Rethrow!
    }
}

§ 22Performance Considerations

AOP adds overhead. Each proxied method call goes through the proxy mechanism. In most applications this is negligible (microseconds), but improper use can cause real problems.

Proxy Overhead Breakdown

  • JDK Proxy call overhead: ~0.5–2μs per method call
  • CGLIB Proxy call overhead: ~0.2–1μs per method call (faster invocation)
  • Pointcut matching: Cached after first evaluation — negligible
  • Heavy advice execution: Depends entirely on what you do inside advice
⚠️ Performance Anti-Patterns
  • Broad pointcuts on hot paths: execution(* *.*(..)) matches everything including framework internals
  • Expensive operations in @Before/@After: Database calls, HTTP requests, synchronous I/O
  • Unnecessary @Around: Use the lightest advice type that meets your need
  • Serializing large objects in logging advice: args.toString() on a 100KB object in every method
Performance Best Practices
// ✅ Be specific — narrow pointcuts
@Pointcut("execution(* com.example.service.*.*(..))")
// NOT: execution(* *.*(..))

// ✅ Use @annotation for opt-in performance monitoring
@Around("@annotation(com.example.annotation.Monitored)")
// Only annotated methods are intercepted

// ✅ Use log level checks to avoid expensive string formatting
@Before("serviceLayer()")
public void logEntry(JoinPoint jp) {
    if (log.isDebugEnabled()) { // Check BEFORE building the string
        log.debug("Entering: {}", jp.getSignature().toShortString());
    }
}

// ✅ Async advice for non-critical operations
@AfterReturning("serviceLayer()")
public void recordMetricsAsync(JoinPoint jp) {
    // Submit to queue, don't block the main thread
    metricsQueue.offer(new MetricEvent(jp.getSignature().getName()));
}

// ✅ Use @After instead of @Around when you don't need the return value
// @After is lighter-weight than @Around

Where AOP Fits in Enterprise Applications

📝
Logging & Tracing Centralized logging of all service/controller calls with correlation IDs, execution times, and structured output. AOP is perfect — logging is the canonical cross-cutting concern.
📊

❌ Avoid AOP for These

  • Data transformation — Mapping DTOs, formatting responses — keep in service layer.
  • Complex conditional logic — If you need complex if/else branching, it belongs in code, not an aspect.
  • Private API concerns — AOP can't intercept private methods. If you need to, rethink the design.
  • Performance-critical hot paths — Avoid broad AOP on methods called millions of times per second.

Weaving Types Explained

⏰
Runtime Weaving (Spring AOP) Proxies are created at application startup. No bytecode manipulation. Easy to set up. Most flexible for Spring applications. Slight overhead per call.
🔨

§ 25Interview Questions & Answers

Beginner Level

BeginnerWhat is AOP and why is it used?
AOP (Aspect-Oriented Programming) is a paradigm for separating cross-cutting concerns from business logic. It's used because behaviors like logging, security, and transactions appear across many unrelated classes. Without AOP, this code is duplicated everywhere. AOP centralizes these concerns in Aspects, keeping business classes clean and focused.
BeginnerWhat is a cross-cutting concern? Give examples.
A cross-cutting concern is functionality that spans multiple application modules and doesn't cleanly fit into a single class. Examples: Logging (needed in every service), Transaction management (needed in every DB operation), Security/authentication (needed at every endpoint), Caching (applicable to many service methods), Auditing (track changes across entities), Performance monitoring (measure every operation).
BeginnerExplain the difference between JoinPoint and Pointcut.
A JoinPoint is a specific point in program execution (in Spring AOP, always a method execution) where advice could be applied. Think of it as a potential interception point. A Pointcut is an expression that selects which JoinPoints will actually have advice applied. JoinPoint = every method call that exists. Pointcut = the filter that picks which ones to intercept.
BeginnerWhat annotations do you need to create an Aspect?
@Aspect marks the class as containing AOP advice logic. @Component (or @Bean) registers it as a Spring-managed bean. Both are required. @Aspect alone is not enough — Spring won't find the class without @Component.
BeginnerWhat are the 5 types of advice?
@Before — Runs before the method. Cannot modify return value. @After — Runs after the method, whether success or exception (like finally). @AfterReturning — Runs only on successful return. Can access the return value. @AfterThrowing — Runs only when an exception is thrown. Can access the exception. @Around — Wraps the entire method. Full control: can skip execution, modify args/return value, handle exceptions.
BeginnerWhat is Weaving?
Weaving is the process of applying aspects to target objects to create advised proxy objects. Spring AOP does this at runtime using JDK Dynamic Proxies or CGLIB. When the application context starts, Spring identifies beans matched by pointcuts and creates proxy wrappers around them.

Intermediate Level

IntermediateWhat is the difference between JDK Dynamic Proxy and CGLIB?
JDK Dynamic Proxy requires the target class to implement at least one interface. It creates a proxy that implements the same interface. CGLIB creates a subclass of the target class and overrides non-final methods. Spring Boot 2.x+ defaults to CGLIB even when interfaces exist. CGLIB cannot proxy final classes or final methods. JDK proxies are interface-bound, so you can only call proxy methods defined in the interface.
IntermediateExplain the execution() pointcut expression syntax.
Format: execution([modifiers] [return-type] [declaring-type].[method-name]([params]) [throws]). The * wildcard matches any single element. The .. wildcard in params means "zero or more arguments"; in package names it means "this package and all subpackages". Example: execution(public * com.example.service..*.*(..)) matches any public method in any class in the service package or its sub-packages, with any return type and any arguments.
IntermediateWhat is the self-invocation problem in Spring AOP?
When a method in a Spring bean calls another method in the same class using this, it bypasses the proxy. Since AOP intercepts are applied at the proxy level, any advice on the called method is silently skipped. This affects @Transactional, @Async, and all custom AOP. Fix: inject the bean into itself (@Autowired @Lazy private MyService self;) or use ApplicationContext.getBean() to get the proxy version.
IntermediateHow does @Transactional work under the hood?
@Transactional is implemented as a Spring AOP aspect (specifically a MethodInterceptor). When you annotate a method, Spring creates a proxy. When the method is called, Spring's TransactionInterceptor runs first: opens a transaction (via PlatformTransactionManager), calls the real method, then commits on success or rolls back on RuntimeException (by default). The proxy is why self-invocation breaks transactions.
IntermediateWhat is the difference between @After and @AfterReturning?
@After is the equivalent of a finally block — it runs whether the method succeeded or threw an exception. It cannot access the return value. @AfterReturning only runs when the method completes normally (no exception). It can access the return value via the returning attribute. Use @After for cleanup that must always happen. Use @AfterReturning when you need to inspect or react to the successful result.
IntermediateWhy must @Around advice declare 'throws Throwable'?
Because pjp.proceed() can throw any checked or unchecked exception from the target method. Since you're wrapping the method call entirely, you must be able to propagate any exception the target might throw. Declaring throws Throwable (or a sufficiently broad exception) allows the compiler to accept the pjp.proceed() call and allows you to rethrow without wrapping.
IntermediateCan you apply multiple aspects to the same method? How is the order controlled?
Yes. Multiple aspects can match the same JoinPoint. Order is controlled with @Order(n) annotation or by implementing Ordered interface. Lower number = higher priority (runs first in @Before, last in @After). If no order is specified, the order is unspecified and can vary. Best practice: assign explicit @Order to all aspects that might conflict.

Advanced Level

AdvancedHow would you implement a distributed tracing system using AOP?
Create an aspect that intercepts all service and controller methods. On entry, check for an incoming trace ID header (X-Trace-ID). If present, store in MDC. If absent, generate a new trace ID. Add this to all outgoing HTTP headers via RestTemplate/WebClient interceptors. The AOP aspect ensures every method execution within a request carries the same trace ID, enabling end-to-end correlation across microservices. This is exactly how Sleuth/Zipkin integration works in Spring.
AdvancedWhat is the difference between Spring AOP and AspectJ? When would you choose AspectJ?
Spring AOP uses runtime proxies (JDK or CGLIB) and can only intercept Spring-managed bean method executions. It uses AspectJ annotation syntax but not the full AspectJ engine. AspectJ uses compile-time or load-time weaving, modifying bytecode directly. AspectJ can intercept field access, static methods, constructors, and private methods. Choose AspectJ when: (1) You need to intercept private methods or static methods, (2) You need to advise non-Spring-managed objects, (3) Performance overhead of proxies is unacceptable on a hot path. For most enterprise applications, Spring AOP is sufficient.
AdvancedHow does @Transactional propagation work with AOP proxies? What happens with REQUIRES_NEW?
REQUIRES_NEW suspends the current transaction and starts a completely new one. This works via AOP: the TransactionInterceptor on the inner proxy checks the propagation setting, calls TransactionManager.suspend(), opens a new connection/transaction, calls the method, commits/rolls back the inner transaction, then resumes the outer transaction. If the inner method is called via self-invocation (using this), the proxy is bypassed and REQUIRES_NEW is completely ignored — both methods run in the same transaction. This is one of the most common production bugs in Spring applications.
AdvancedExplain how you would implement an idempotency key system using AOP.
Create an @Idempotent annotation that takes a key expression (SpEL). The aspect: (1) Extracts the idempotency key from request headers or method args using SpEL evaluation. (2) Checks Redis for a stored result under this key. (3) If found, returns the stored result immediately without calling the method (achieving idempotency). (4) If not found, calls the method, stores the result in Redis with a TTL, and returns it. This pattern is critical for payment and financial APIs where duplicate requests must return the same result without re-executing.
AdvancedHow does Spring's @EnableAspectJAutoProxy work internally?
It registers AnnotationAwareAspectJAutoProxyCreator as a BeanPostProcessor in the Spring context. This BPP runs after every bean is created. For each bean, it checks all registered @Aspect classes to find any pointcuts that match the bean's methods. If a match is found, instead of returning the raw bean, it returns a proxy (JDK or CGLIB) that wraps the bean and applies the matched advice. The proxy is stored in the context, so all @Autowired injections receive the proxy, not the original bean.
AdvancedWhat is the execution order when @Before and @After from multiple aspects fire on the same method?
With two aspects A (Order=1) and B (Order=2) on the same method: @Before from A fires first (lower order = outer layer), then @Before from B, then the method executes, then @After from B, then @After from A. This is analogous to nested try-finally blocks: A wraps B wraps the method. The outer aspect's @Before runs first; its @After runs last. This is the "Russian doll" model of AOP advice application.
AdvancedCan AOP affect the constructor of a Spring bean? How?
Spring AOP cannot intercept constructors because proxies are created after the bean is instantiated. The proxy wraps the existing object — it doesn't replace the constructor. If you need constructor-level interception, you need compile-time or load-time AspectJ weaving. In Spring, constructor-related cross-cutting concerns are typically handled via @PostConstruct for initialization logic, or by abstracting the construction through factory methods that can be proxied.
AdvancedHow would you debug when an AOP advice is not firing?
Checklist: (1) Is the Aspect class annotated with both @Aspect AND @Component? (2) Is the method private? AOP can't intercept private methods. (3) Is the method final? CGLIB can't override final methods. (4) Is it a self-invocation? this.method() bypasses proxy. (5) Is the pointcut expression correct? Add logging to confirm — temporarily use execution(* *.*(..)) to match everything. (6) Is the class managed by Spring? AOP only applies to Spring-managed beans. (7) Enable Spring AOP debug: logging.level.org.springframework.aop=DEBUG.

SmartLoggingAspect.java — Full Implementation
@Aspect
@Component
@Slf4j
public class SmartLoggingAspect {

    @Autowired(required = false)
    private List<MethodCallListener> listeners = Collections.emptyList();

    // Pointcut 1: @Logged on class or method
    @Pointcut("@within(com.example.annotation.Logged) || @annotation(com.example.annotation.Logged)")
    public void loggedMethods() {}

    // Pointcut 2: Exclude @NoLog methods
    @Pointcut("!@annotation(com.example.annotation.NoLog)")
    public void notSuppressed() {}

    // Combined
    @Pointcut("loggedMethods() && notSuppressed()")
    public void intercept() {}

    @Around("intercept()")
    public Object smartLog(ProceedingJoinPoint pjp) throws Throwable {
        // Get annotation (method-level overrides type-level)
        MethodSignature sig = (MethodSignature) pjp.getSignature();
        Method method = sig.getMethod();
        Logged logged = method.getAnnotation(Logged.class);
        if (logged == null) {
            logged = pjp.getTarget().getClass().getAnnotation(Logged.class);
        }

        String traceId = MDC.get("traceId");
        if (traceId == null) {
            traceId = UUID.randomUUID().toString().substring(0, 8);
            MDC.put("traceId", traceId);
        }

        String className = pjp.getTarget().getClass().getSimpleName();
        String methodName = sig.getName();
        Object[] args = logged.logArgs() ? pjp.getArgs() : new Object[]{"[HIDDEN]"};

        // Build log prefix
        String prefix = String.format("[%s] %s.%s()", traceId, className, methodName);

        logAtLevel(logged.level(), "→ {} args={}", prefix, formatArgs(args));

        long start = System.nanoTime();
        MethodCallRecord.MethodCallRecordBuilder record = MethodCallRecord.builder()
            .traceId(traceId).className(className).methodName(methodName)
            .arguments(args).timestamp(Instant.now());

        try {
            Object result = pjp.proceed();
            long ms = (System.nanoTime() - start) / 1_000_000;
            Object logResult = logged.logResult() ? result : "[HIDDEN]";

            if (logged.logTime() && ms > logged.warnThresholdMs()) {
                log.warn("← {} SLOW {}ms result={}", prefix, ms, logResult);
            } else {
                logAtLevel(logged.level(), "← {} {}ms result={}", prefix, ms, logResult);
            }

            record.success(true).returnValue(result).durationMs(ms);
            notifyListeners(record.build());
            return result;

        } catch (Exception ex) {
            long ms = (System.nanoTime() - start) / 1_000_000;
            log.error("✗ {} FAILED in {}ms: [{}] {}", prefix, ms,
                ex.getClass().getSimpleName(), ex.getMessage());
            record.success(false).exception(ex).durationMs(ms);
            notifyListeners(record.build());
            throw ex;
        }
    }

    private void logAtLevel(String level, String format, Object... args) {
        switch (level.toUpperCase()) {
            case "DEBUG": log.debug(format, args); break;
            case "WARN":  log.warn(format, args); break;
            case "ERROR": log.error(format, args); break;
            default:      log.info(format, args);
        }
    }

    private String formatArgs(Object[] args) {
        if (args == null || args.length == 0) return "[]";
        return Arrays.stream(args)
            .map(a -> a == null ? "null" : sanitize(a.toString()))
            .collect(Collectors.joining(", ", "[", "]"));
    }

    private String sanitize(String s) {
        // Redact passwords, tokens
        if (s.toLowerCase().contains("password") || s.toLowerCase().contains("token")) {
            return "***";
        }
        return s.length() > 100 ? s.substring(0, 100) + "..." : s;
    }

    private void notifyListeners(MethodCallRecord record) {
        listeners.forEach(l -> {
            try { l.onMethodCall(record); }
            catch (Exception e) { log.error("Listener error", e); }
        });
    }
}

§ 27Best Practices

✅
Use the least powerful advice type If you only need to log method entry, use @Before — not @Around. @Around has more power but also more risk (forgetting to return the result, swallowing exceptions). Choose the simplest advice that solves the problem.
✅
Keep pointcuts specific and narrow Avoid execution(* *.*(..)). Use package-specific expressions. Broad pointcuts can inadvertently intercept Spring internals, framework beans, and cause performance issues or subtle bugs.
✅
Name and reuse pointcuts Define pointcuts in a centralized class (PointcutLibrary). Reuse with fully-qualified references. This prevents duplicated pointcut strings and makes global changes easy.
✅
Always return proceed() result in @Around Never discard the return value. Even if you don't care about it, the caller does. Returning null silently breaks all callers of that method.
✅
Always rethrow in @Around unless intentionally handling Swallowing exceptions silently is worse than crashing. If you catch and don't rethrow, document clearly why. Usually, log + rethrow is the right pattern.
✅
Use @Order to make aspect ordering explicit Don't rely on undefined ordering between aspects. Assign explicit @Order values. Security aspects should run first (order=1), then audit, then logging.
✅
Use custom annotations for opt-in AOP Prefer @annotation() pointcuts for explicit opt-in behavior (e.g., @LogExecutionTime) over broad package-matching for anything beyond universal infrastructure concerns. This makes the behavior visible in the code.
✅
Be careful with expensive operations in advice Advice runs synchronously in the method's call chain. If your advice does a database call or HTTP request, it adds latency to every intercepted method. Use async queues for non-critical operations.
✅
Don't put business logic in aspects Aspects are for infrastructure concerns. Business rules belong in services. If you find yourself putting complex conditional business logic in an aspect, it's a design smell.
✅
Test your aspects Write integration tests that verify advice fires correctly. Test: (1) advice fires for matched methods, (2) advice doesn't fire for non-matched methods, (3) exception propagation works correctly, (4) return values are not modified unexpectedly.

@Around("yourPointcut()")
public Object aroundAdvice(ProceedingJoinPoint pjp) throws Throwable {
    // BEFORE
    long start = System.currentTimeMillis();
    
    try {
        Object result = pjp.proceed(); // Call the actual method
        // AFTER RETURNING
        return result; // Always return!
    } catch (Exception ex) {
        // ON EXCEPTION
        throw ex; // Always rethrow (unless intentionally swallowing)!
    } finally {
        // ALWAYS
        long elapsed = System.currentTimeMillis() - start;
    }
}

§ 30Final Summary

Key Takeaways

🎯
AOP = Modularizing what OOP can't AOP doesn't replace OOP — it complements it. OOP organizes objects and behavior. AOP organizes cross-cutting concerns that span multiple objects. Together, they produce clean, maintainable enterprise code.
🔑
The proxy is the key mental model Everything in Spring AOP flows from the proxy concept. When you inject a bean, you get a proxy. When the proxy's method is called, advice fires. When advice calls proceed(), the real method runs. This chain explains all AOP behavior — including its limitations.
⚡
You're already using AOP @Transactional, @Cacheable, @Async, @PreAuthorize — these are Spring AOP aspects. Understanding AOP means understanding how half of Spring's most important features work internally. This makes you a far more effective Spring developer.
🛡️
Know the limitations Private methods, final methods, self-invocation — these are the three failure modes that catch every developer at least once. Know them before they hit you in production.
🚀
Custom annotations = the killer pattern The most powerful real-world use of AOP is creating custom annotations (@LogExecutionTime, @AdminOnly, @RateLimit, @Idempotent) and backing them with aspects. This gives you Spring-native declarative programming for any concern.
📐
Pointcuts are a skill Writing correct, narrow pointcut expressions separates beginners from experienced AOP users. Practice them. Know the difference between * and .., between execution() and within(), and how to combine them with && || !.

Mental Models to Internalize

  1. The Glass Pane Model — AOP is a transparent layer over your application. Business code stays clean underneath.
  2. The Russian Doll Model — Multiple aspects nest around a method. Outer aspects see inner aspects' effects. Order matters.
  3. The Proxy Interception Model — You never call the real object directly (after AOP is applied). You always call the proxy first.
  4. The Declarative Model — AOP enables declarative programming. You declare what concerns apply to what methods. The framework handles the how.
